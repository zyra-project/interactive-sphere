# Installing Terraviz on macOS

If you downloaded the macOS DMG and macOS shows **"Terraviz is
damaged and can't be opened. You should move it to the Trash"**, the
app isn't actually damaged — this is macOS Gatekeeper rejecting the
bundle because it isn't yet signed with an Apple **Developer ID
Application** certificate. Until signing is in place (the release
workflow is already wired up to consume the cert; see
[`DESKTOP_APP_PLAN.md`](DESKTOP_APP_PLAN.md) §Setup steps), use one
of the two workarounds below. Both are one-time per download.

> **Apple Silicon only.** Terraviz currently ships an
> `aarch64-apple-darwin` build (M1/M2/M3/M4). The DMG won't run on
> Intel Macs regardless of signing. If you need Intel support, open
> an issue.

---

## Option 1: Terminal (most reliable)

Drag `Terraviz.app` out of the mounted DMG into `/Applications` (or
wherever you keep apps), then run:

```bash
xattr -dr com.apple.quarantine /Applications/Terraviz.app
```

Launch the app normally. If the first launch shows a "can't verify
the developer" prompt instead, click **Open**.

### What the command does

Every file downloaded from the internet picks up a
`com.apple.quarantine` extended attribute. Gatekeeper reads that
attribute to decide whether to verify a signature on first launch;
stripping it tells macOS you trust this specific copy of the app.
The `-r` flag makes it recursive so it covers the nested binaries
inside the `.app` bundle.

This is a per-bundle change. macOS will still set the attribute on
the next download.

---

## Option 2: System Settings (no terminal)

1. Drag `Terraviz.app` from the DMG to `/Applications`.
2. Double-click to launch — the "damaged" dialog appears. Click
   **Cancel**.
3. Open **System Settings** → **Privacy & Security**.
4. Scroll to the **Security** section. You should see a line that
   reads `"Terraviz" was blocked from use because it is not from an
   identified developer.`
5. Click **Open Anyway**.
6. Re-launch Terraviz. A confirmation dialog appears; click
   **Open**.

If the "Open Anyway" button doesn't appear, your macOS version is
suppressing it for the "damaged" verdict specifically — fall back
to Option 1.

---

## Why this is necessary

Apple requires macOS apps distributed outside the App Store to be
signed with a **Developer ID Application** certificate and
notarized by Apple's notary service. Without those, the binary
runs only on machines that have explicitly bypassed Gatekeeper for
it — which is what the workarounds above do.

The certificate must be generated by the Apple Developer Program
**Account Holder** for our organization. The release workflow is
already wired up to consume it and submit the bundle to notarytool
on every tagged release. Once the cert is added to the repo's
GitHub Actions secrets, future downloads will open cleanly without
either workaround.

---

## Verifying a notarized build (once signing lands)

If you want to check that a future release is properly notarized
before you trust it, on a Mac with the DMG mounted and the `.app`
copied to `/Applications`, run:

```bash
codesign --verify --deep --strict /Applications/Terraviz.app
spctl --assess --type execute /Applications/Terraviz.app
```

The first command should return silently (no output = success).
The second should print
`/Applications/Terraviz.app: accepted` followed by
`source=Notarized Developer ID`. If you see `source=Unsigned` or
`source=Unnotarized Developer ID`, the build is not the final
notarized artifact.
